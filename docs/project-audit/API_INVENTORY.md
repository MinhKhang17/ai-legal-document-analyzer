# API inventory

## Counting and conventions

Static inspection found **212 Spring handler methods in 31 controllers** and **25 FastAPI handler functions**. Spring class aliases (`/api/v1/workspaces` + `/api/workspaces`, `/api/v1/subscriptions` + `/api/subscription`) and the admin feedback alias expose more concrete URL/method combinations than handler count. Responses normally use `ApiResponseDTO<T>`; pages use `PageResponse<T>`; binary downloads/exports use `byte[]`/resource responses. `@Valid` DTOs enforce Jakarta annotations; multipart endpoints additionally enforce file policy. Unless listed public/internal, Spring requires JWT, and method-level role rules apply.

Status codes and database effects are determined in controllers/services/global handlers; common mismatches are summarized after the inventory.

## Known mismatches and API risks

- Public Spring matchers include document downloads, internal callbacks and admin bulk ingest; FastAPI has no inter-service authentication. This is the highest-priority contract/security issue.
- `/api` legacy ticket routes coexist with `/api/v1/legal-tickets`; workspace/subscription aliases double concrete surfaces and increase drift risk.
- The frontend route for shared ticket conversation is authenticated although its API is token-public.
- FastAPI validation naturally returns 422, while Spring generally wraps errors in JSON; cross-service translation must preserve a stable machine code.
- Spring missing routes are handled through global exception logic, but static-resource `NoResourceFoundException` behavior must remain covered so API 404s are JSON.
- Subscription restrictions are service-enforced, not URL-enforced: token/storage quotas, Free support ticket count, Premium expert access, file size and session attachment limits.

FLOW 05 - ADMIN DUYỆT VÀ PHÂN CÔNG TICKET

Mục tiêu:
- Cho admin xem toàn bộ legal ticket, lọc theo status/risk, xem AI summary/chat/files và phân công hoặc từ chối ticket.

Actor:
- `ADMIN`

Điểm bắt đầu:
- Route `/admin/tickets`
- Route `/admin/tickets/:ticketId`

Điều kiện tiên quyết:
- User role admin

Luồng chính:
1. `AdminTicketsPage` tải danh sách ticket với filter status và risk.
2. FE gọi `getAdminLegalTickets(page, size, status?, riskLevel?)`.
3. Admin chọn một ticket để mở detail.
4. `AdminTicketDetailPage` tải song song:
5. ticket detail
6. admin AI summary
7. chat history
8. user files
9. AI assessment
10. AI summary/citations
11. danh sách user để chọn expert
12. Admin chọn expert và gọi assign hoặc reassign.
13. Admin có thể nhập reject reason và từ chối ticket.

Sơ đồ text:
- `admin list -> filter -> open detail -> inspect evidence -> assign/reassign/reject`

Màn hình / component:
- `frontend/src/pages/admin/AdminTicketsPage.tsx`
- `frontend/src/pages/admin/AdminTicketDetailPage.tsx`

Dữ liệu / state:
- `tickets`, `statusFilter`, `riskLevel`
- detail: `ticket`, `ticketSummary`, `chatMessages`, `files`, `assessment`, `aiSummary`, `citations`
- assignment: `experts`, `selectedLawyerId`, `adminNote`
- rejection: `rejectReason`

Validation:
- Assign/reassign phải có `selectedLawyerId`
- Reject phải có `reason`
- Danh sách expert hiện lọc phía FE: `user.active && user.role === "EXPERT"`

API / service tích hợp:
- FE:
- `frontend/src/services/legalTicket.service.ts`
- `frontend/src/services/user.service.ts`
- BE:
- `GET /api/v1/admin/tickets`
- `GET /api/v1/admin/tickets/{id}`
- `GET /api/v1/admin/tickets/{id}/summary`
- `GET /api/v1/admin/tickets/{id}/chat-history`
- `GET /api/v1/admin/tickets/{id}/files`
- `POST /api/v1/admin/tickets/{id}/assign-lawyer`
- `POST /api/v1/admin/tickets/{id}/reassign-lawyer`
- `POST /api/v1/admin/tickets/{id}/reject`

UI states:
- Dashboard thống kê tổng ticket/open ticket/current filter
- Detail có block AI summary, AI assessment, chat history, file list, citations

Edge cases:
- Nếu `getUsers()` fail, admin vẫn mở được detail nhưng không assign được
- `hasAssignedLawyer` ảnh hưởng nút assign/reassign
- Rule reassign khi `force_reassign` cần backend chốt

Business rules:
- Admin nhìn thấy toàn hệ thống, không chỉ ticket của mình
- Assign và reassign dùng cùng payload gần giống nhau
- Reject là bước review của admin trước khi expert xử lý

Quyền:
- Tất cả endpoint trong flow này đều `hasRole('ADMIN')`

File nguồn tham chiếu:
- FE:
- `frontend/src/pages/admin/AdminTicketsPage.tsx`
- `frontend/src/pages/admin/AdminTicketDetailPage.tsx`
- `frontend/src/services/legalTicket.service.ts`
- `frontend/src/services/user.service.ts`
- BE:
- `backend/src/main/java/com/analyzer/api/controller/admin/AdminTicketManagementController.java`

Điểm chưa rõ:
- `CHƯA XÁC ĐỊNH`: có pagination/size thực tế nào business muốn dùng ở admin list.
- `CHƯA XÁC ĐỊNH`: admin có thể undo reject hay không.

Checklist FE:
- Nên phân biệt rõ `assign` lần đầu và `reassign`.
- Cần hiển thị user role/name/email khi chọn expert để tránh chọn nhầm.
- Nên disable action theo status ticket nếu backend có state machine chặt.

Tiêu chí hoàn thành:
- Admin xem được list và detail.
- Admin assign/reassign/reject được.
- Admin đọc được chat history, file, AI summary/citation liên quan.

HIỆN TRẠNG:
- Flow này đã có route, page, service và controller đồng bộ tương đối tốt.

SUY LUẬN:
- Đây là lớp trung gian giữa AI suggestion/customer request và expert execution.

ĐỀ XUẤT:
- FE nên làm rõ visual state theo `unassigned / assigned / rejected / resolved` để thao tác admin nhanh hơn.
